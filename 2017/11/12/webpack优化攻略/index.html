webpack优化攻略

开发环境

开发过程中使用最多的方式就是HMR( hot module replacement )，热替换的概念是指 webpack 会监听源代码的修改，当监听文件改变时，会以 json 数据的形式通知客户端进行检查更新，客户端发送请求获取更新后的代码并合并到当前代码中，实现不刷新页面的情况下进行局部更新。

生产环境

配置CommonsChunkPlugin

CommonsChunkPlugin插件的功能是在多入口的情况下将打包过程中的公共模块抽离到一个独立的文件，以减小最后 bundle 文件的体积。

加速文件搜索

配置 resolve.modules

Webpack的 resolve.modules 配置模块库（即 node_modules）所在的位置，在 js 里出现类似 import ‘vue’ 这样即不是相对也不是绝对路径的写法时，会去 node_modules 目录下找。但是默认的配置，会采用向上递归搜索的方式去寻找，但通常项目目录里只有一个 node_modules，且是在项目根目录，为了减少搜索范围，可以直接写明 node_modules 的全路径， src 目录同理；

设置 test & include & exclude

对于include，更精确指定要处理的目录，这可以减少不必要的遍历，从而减少性能损失。同样，对于已经明确知道的，不需要处理的目录，则应该予以排除，从而进一步提升性能。假设你有一个第三方组件的引用，它肯定位于 node_modules，通常它将有一个 src 和一个 dist 目录。如果配置 Webpack 来排除 node_modules，那么它将从 dist 已经编译的目录中获取文件。否则会再次编译它们。故而，合理的设置 include & exclude，将会极大地提升 Webpack 打包优化速度

使用并行压缩插件 webpack-parallel-uglify-plugin

webpack自带的js压缩插件是UglifyJsPlugin，该压缩插件是单线程压缩，速度比较慢，webpack-parallel-uglify-plugin 这个插件可以并行运行多个 UglifyJs 插件，充分利用 CPU 资源。

替换压缩插件前后 wmda 项目的打包时间缩短了8s左右。

配置externals

Webpack 可以配置 externals 参数来将依赖的库指向全局变量，从而不再打包这个库，然后用cdn将该库引入至全局。

例如，项目中 import 了 react 库，此时可以配置 externals 将 react 指向 window.React 这个全局变量，然后用 script 标签将 react.min.js 引入页面保证 window.React 存在。

这时 webpack 将不再打包 react 库，所以打包速度大幅提升。但是问题没有这么简单，如果此时我们又 import 了一个 react 子库 react-bootstrap 到项目中，会发现 webpack 又把 react 整个库给打包了。

原因是，打包 react-bootstrap 时，该库依赖了 react 主库，webpack此时不知道这个依赖入口就是 externals 中配置的 react 模块，所以又傻傻的打包了一遍。可能你会说，把 react-bootstrap 这个库也配置到 externals 里不就行了吗？但问题并不是所有的第三方库都提供了用于生产环境的文件，也就是说没有提供 react-bootstrap.min.js 这个文件。

解决方法就是使用下面说的 DllPlugin 和 DllReferencePlugin 。

使用DllPlugin和DllReferencePlugin

dll（动态链接库）是 window 系统中一种常见的思想。一个 dll 文件，就是一个依赖库，它本身不能运行也不会修改，是给其他模块或者业务代码引用的。

DllPlugin 和 DllReferencePlugin 需要配合使用，它们提供了一种用拆分软件包预构建的打包方法。其原理是，将指定的第三方NPM包模块提前构建，然后通过页面引入。这不仅能够使得打包后的公共模块 vendor.js 文件体积大幅度减小，同时，也极大的提高了构建速度。
